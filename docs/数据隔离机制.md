# RAG 数据隔离机制

> 文档版本：2026-04-03
> 适用：AI-Studio RAG 模块（Knowledge Base + DenseStore/SparseStore）

---

## 概述

RAG 模块实现数据隔离的核心思路是**两层隔离**：

1. **ORM 层** — `user_id` 归属验证，阻止跨用户 KB 访问
2. **向量存储层** — `kb_id` 作为 metadata filter，确保 KB 间数据不串扰

---

## 一、索引（入库）时的隔离

```
service.index_file(kb_id, file_id, ...)
  → rag_service.index_document(
        kb_id=kb.id,
        file_id=file_id,
        metadata={"kb_id": kb.id, "file_id": file_id},
    )
      → 构建 DocumentUnit(kb_id=kb.id, file_id=file_id, ...)
          → dense_store.add_documents(doc, embedding)
          → sparse_store.add_documents(doc)
```

`kb_id` 作为 `DocumentUnit.metadata` 的一部分，**同时存了两份**：

| 存储位置 | 字段 | 说明 |
|----------|------|------|
| `pg_chunks` 表 | `kb_id` 列 | 普通索引列，可用于 SQL WHERE 过滤 |
| `pg_chunks.metadata` JSONB | `kb_id` key | 用于 metadata_filter 精确过滤 |

入库本身没有 `user_id` 过滤 — 因为入库是通过 `kb` 对象操作的，而 `kb` 在 ORM 层已经有 `user_id` 归属。

## 二、检索时的隔离

### 1. ORM 层 — `kb_id` 的归属验证

```python
# service.py:286
await self.get_kb(kb_id, user_id)   # 验证 kb 属于该 user

# service.py:305  — 每个 search_kb_id 都验证
search_kb = await self.get_kb(search_kb_id, user_id)
```

`get_by_id(kb_id, user_id)` 的 SQL 是：

```sql
WHERE id = :kb_id AND user_id = :user_id
```

如果用户 A 传 `kb_id=B`（属于用户 B），这个查询返回 `None`，抛 `NotFoundException`。**这一层确保了用户只能操作自己的 KB。**

### 2. 向量存储层 — `kb_id` 过滤

```python
# service.py:313
results_with_scores = await search_rag_service.retrieve(
    query=request.query,
    top_k=config.top_k,
    vector_weight=config.vector_weight,
    metadata_filter={"kb_id": search_kb.id},  # ← 关键
)
```

传入的 `metadata_filter` 在 `PGDenseStore` 和 `PGSparseStore` 的 SQL 中被应用：

```sql
-- PGDenseStore.retrieve() (stores/pg_dense.py:67-92)
SELECT ..., 1 - (embedding <=> :embedding) AS score
FROM pg_chunks
WHERE metadata ->> 'kb_id' = :meta_0   -- 只返回该 kb_id 的文档
  AND ...
ORDER BY embedding <=> :embedding LIMIT :top_k
```

即：**入库时 `kb_id` 被嵌入文档的 metadata 中，检索时通过 `metadata_filter` 精确过滤。**

## 三、用户 A 的 `[kb1, kb2]` vs 用户 B 的 `[kb3, kb4]`

### 用户 A 检索自己的知识库

```
retrieve(kb_ids=[kb1, kb2], user_id=A)
  ├─ ORM 层验证: get_kb(kb1, A) ✓, get_kb(kb2, A) ✓
  ├─ per KB 创建 RAGRetrievalService
  └─ per KB 执行检索:
       metadata_filter={"kb_id": kb1}  → 只返回 kb1 的文档
       metadata_filter={"kb_id": kb2}  → 只返回 kb2 的文档
  结果合并，按 score 排序返回
```

### 用户 A 试图访问用户 B 的知识库

```
retrieve(kb_ids=[kb3], user_id=A)  # kb3 属于用户 B
  └─ ORM 层: get_kb(kb3, A) → None → NotFoundException
     ↑ 业务层直接拒绝，根本不会调用向量存储
```

### 隔离检查表

| 场景 | ORM 层拦截 | 向量存储层拦截 |
|------|-----------|--------------|
| 用户 A 检索自己的 `kb1` | `get_kb(kb1, A)` ✓ | `metadata_filter={"kb_id": kb1}` ✓ |
| 用户 A 传入 `kb3`（属于 B） | `get_kb(kb3, A)` → None → **拒绝** | 不执行 |
| 用户 A 传不存在的 `kb_id` | `get_kb(kb_none, A)` → None → **拒绝** | 不执行 |

## 四、ChromaDB 隔离

`ChromaDenseStore` 使用 collection 级别的隔离：

```python
# service_factory.py:40-44
return ChromaDenseStore(
    persist_directory=settings.CHROMA_PERSIST_DIR,
    collection_name=kb.collection_name,  # ← 每个 KB 独立 collection
    user_id=kb.user_id,
)
```

collection 名称在 KB 创建时生成：

```python
# service.py:58-60
collection_name = f"kb_{user_id}_{data.name.lower().replace(' ', '_')}_{uuid.uuid4().hex[:8]}"
```

ChromaDB 本身通过 collection 名称实现物理隔离，`user_id` 存在 collection metadata 中作为二次验证。

## 五、关键代码路径

### 索引路径

```
KnowledgeBaseService.index_file()
  → create_rag_index_service(kb)          # service_factory.py
      → PGDenseStore.add_documents()        # kb_id 写入 metadata
      → PGSparseStore.add_documents()       # kb_id 写入 metadata
```

### 检索路径

```
KnowledgeBaseService.retrieve()/rag()
  → get_kb(kb_id, user_id)                 # ORM 层 user_id 验证
  → create_rag_retrieval_service(search_kb)
      → PGDenseStore.retrieve(
            metadata_filter={"kb_id": search_kb.id}  # 向量存储层过滤
        )
      → PGSparseStore.retrieve(
            metadata_filter={"kb_id": search_kb.id}
        )
```

## 六、总结

```
两层隔离模型：

ORM 层（PostgreSQL）
  └── WHERE kb_id = ? AND user_id = ?  ← 阻止跨用户访问

向量存储层（pgvector / ChromaDB）
  └── metadata_filter: {"kb_id": ?}     ← KB 间数据不串扰
```

- **入库**：通过 `kb` 对象操作，`kb_id` 自动注入 metadata
- **检索**：先过 ORM `user_id` 验证，再在向量存储层用 `kb_id` 过滤
- **跨租户防护**：ORM 层直接拒绝，物理上不执行向量查询
