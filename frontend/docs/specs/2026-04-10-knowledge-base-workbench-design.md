# 知识库工作台设计方案

## 背景

当前知识库前端已经具备基础列表页与详情页骨架，详情页采用 `Overview / Files / Retrieve / Chat` 四个 tab 组织。但现阶段仍存在明显问题：

- `Overview` 更像摘要卡片，不是正式配置页
- `Retrieve` 和 `Chat` 结果展示仍然偏调试态，离正式工具产品体验有距离
- 用户进入知识库后，缺少“这是一个可以持续操作和验证的工作台”感受

结合当前项目定位、后端 API 现状以及同类工具操作逻辑，知识库模块应采用：

**一个知识库 = 一个工作台页面**

即：

- 列表页负责入口、创建和筛选
- 详情页负责围绕单个知识库进行配置、文件管理、检索调试和问答验证

## 方案结论

推荐保留当前详情页 tab 结构，但将 `Overview` 升级为正式配置工作区，而不是再单独拆一个纯配置页。

### 为什么不拆独立配置页

- 知识库不是一次性填写配置后就结束的资源，而是需要持续调试和验证的工作对象
- 用户真实操作顺序通常是：建库 -> 配置 -> 上传文件 -> 索引 -> 检索验证 -> RAG 对话验证
- 如果配置页与工作页完全分离，会割裂知识库的使用节奏

### 为什么当前方案更合适

- 与现有路由、API 和组件结构最兼容
- 更符合 Cherry Studio / RAGFlow 这类工具的工作台心智
- 后续扩展检索参数、引用片段、问答记录时不需要再推翻页面结构

## 目标

本阶段的目标是把知识库详情页升级成一个真正可用的知识库工作台：

- `Overview`：正式配置页
- `Files`：文件上传、状态查看、索引管理
- `Retrieve`：正式版检索调试页
- `Chat`：正式版知识库问答页

## 非目标

本阶段不做以下内容：

- 多知识库混合问答工作流
- 流式 RAG Chat
- 复杂的文件树、批量文件操作
- 高级检索实验台（如多组参数对比、Rerank 解释视图）
- 本地离线缓存和桌面原生文件系统增强

## 用户心智

### 核心对象

- 知识库列表中的一项：一个知识库资产
- 知识库详情页：该资产的工作台

### 用户操作流

1. 用户进入知识库列表页
2. 点击某个知识库进入详情页
3. 在 `Overview` 中维护基础配置
4. 在 `Files` 中上传和索引文件
5. 在 `Retrieve` 中验证检索结果是否合理
6. 在 `Chat` 中验证最终问答效果

这意味着详情页不能只展示“静态详情”，而要服务于持续操作。

## 页面结构

### 列表页

列表页继续承担：

- 知识库列表浏览
- 创建知识库
- 快速进入知识库工作台

本阶段列表页不做重构，只做必要的样式和交互补充。

### 详情页总结构

详情页保留四个 tab：

- `Overview`
- `Files`
- `Retrieve`
- `Chat`

但四个 tab 的职责重新定义如下。

## Tab 设计

### Overview：正式配置页

`Overview` 从摘要页升级为正式配置页，分三块内容：

1. 基础信息
- 名称
- 描述

2. 切片与检索默认参数
- `chunk_size`
- `chunk_overlap`
- `top_k`

3. 关联能力摘要
- 当前默认 Chat Model
- 文件数量
- 最近更新时间

交互要求：

- 使用 RHF + Zod 做正式表单
- 支持“保存配置”
- 保存成功后刷新当前详情 query
- 参数变更后可自然衔接到 `Retrieve` 和 `Chat`

视觉要求：

- 不再是四张指标卡简单展示
- 应该更像一个“配置工作区”
- 右侧可保留少量只读摘要卡片，但左侧必须是正式表单

### Files：文件与索引

`Files` 保持当前方向，但升级成可用态：

- 顶部保留上传区
- 下方展示文件列表
- 每个文件显示：
  - 文件名
  - 状态
  - 文件大小
  - 更新时间
  - 重新索引入口

后续可扩展但当前先不做的内容：

- 删除文件
- 批量索引
- 文件预览

### Retrieve：正式版检索调试页

当前 `Retrieve` 还是“请求框 + 原始列表”思路，本阶段升级为正式检索验证工作区。

结构建议：

1. 左侧输入区
- 查询输入框
- 执行检索按钮
- 可选的检索参数显示区（本阶段先只读展示当前 `top_k`）

2. 右侧结果区
- 每个结果用卡片呈现
- 信息包括：
  - 排名
  - 分数
  - 片段内容
  - 来源标识（如果后端已有可用字段则展示，没有则先不强造）

3. 空态与说明区
- 未检索前给出用途说明
- 无结果时给出“可尝试换关键词/确认文件已索引”

关键点：

- 不再展示调试式 JSON 感
- 强调“检索结果是否符合预期”
- 为 `Chat` 页的问答体验做前置验证

### Chat：正式版知识库问答页

`Chat` 要从“请求 + 回答块”升级成轻量聊天工作区。

结构建议：

1. 顶部上下文条
- 当前知识库名称
- 当前选择的 Chat Model

2. 中间消息区
- 区分用户问题和助手回答
- 助手回答下方展示引用片段或上下文块摘要（如果接口返回）

3. 底部输入区
- 多行输入框
- 发送按钮

4. 辅助区
- 首次空态提示用户可问什么
- 错误态显示清晰提示，而不是原始响应体

关键点：

- 这是“验证该知识库是否真正可回答问题”的页面
- UI 要更像聊天产品，而不是调试表单

## 数据与 API 映射

当前前端已接上的知识库 API 可以继续沿用：

- `GET /v1/knowledge-bases`
- `GET /v1/knowledge-bases/{kb_id}`
- `POST /v1/knowledge-bases`
- `GET /v1/knowledge-bases/{kb_id}/files`
- `POST /v1/knowledge-bases/{kb_id}/files`
- `POST /v1/knowledge-bases/{kb_id}/files/{file_id}/index`
- `POST /v1/knowledge-bases/retrieve`
- `POST /v1/knowledge-bases/rag`

本阶段额外要求：

- 如果后端已有知识库更新接口，则 `Overview` 应接正式保存
- 如果当前还未封装更新 hook，需要先在 `api/endpoints/knowledge-base.ts` 增补

## 前端模块边界

建议按 feature 化方式拆分知识库详情页：

- `features/knowledge-base/detail/overview-form.tsx`
- `features/knowledge-base/detail/files-panel.tsx`
- `features/knowledge-base/detail/retrieve-panel.tsx`
- `features/knowledge-base/detail/chat-panel.tsx`
- `features/knowledge-base/detail/types.ts`
- `features/knowledge-base/detail/schema.ts`

详情页路由本身只负责：

- 读取 `kbId`
- 拉取主 query
- 管理 tab 容器

各 tab 面板组件负责自己的交互和 mutation。

## 状态管理规则

### TanStack Query

用于：

- 知识库详情
- 文件列表
- 检索结果
- Chat 请求结果

### 本地组件状态

用于：

- 当前检索输入
- 当前聊天输入
- 当前上传文件
- 当前选中的 chat model

### Zustand

本阶段不为知识库详情页新增全局 Zustand 状态，避免把本地工作区状态抬升过高。

## 异常与反馈

每个 tab 都必须有：

- loading 态
- 空态
- error 态

前端展示原则：

- 用户看到的是业务可理解提示
- 原始后端 traceback 不应直接暴露
- 表单和操作按钮应有 pending 状态

## 验收标准

完成后应满足：

1. 用户进入知识库详情页后，能够明显感知这是一个“知识库工作台”
2. `Overview` 可作为正式配置页使用，而不是只读摘要页
3. `Retrieve` 结果展示不再有明显 JSON 调试感
4. `Chat` 页具备基础聊天产品形态
5. 文件上传、索引、检索、问答主链路可顺畅联调

## 推荐实施顺序

1. 升级 `Overview` 为正式配置页
2. 重构 `Retrieve` 成正式检索调试页
3. 重构 `Chat` 成正式问答页
4. 补 `Files` 的状态反馈和细节体验
5. 统一知识库详情页的 loading / empty / error 体验

## 风险点

- 如果后端没有知识库更新接口，`Overview` 表单保存会被阻塞
- 如果 RAG 接口返回结构不稳定，`Chat` 的引用片段展示需要做容错
- 如果文件状态字段语义不稳定，`Files` 页的状态文案需先统一

## 结论

知识库模块下一阶段不应另起一套独立配置页，而应继续沿用当前详情页 tab 结构，并将其升级为：

**配置、文件、检索、问答四位一体的知识库工作台。**
