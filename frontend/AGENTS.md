# AGENTS.md — Frontend 长任务开发约束

> 适用于 `frontend/` 目录下的前端持续开发。  
> 目标是在多轮会话中保持设计一致、任务可交接、实现可验证。

---

## 核心原则

你是今天的前端值班工程师。你需要在有限上下文内，优先做清晰、可验证、可交接的工作。

**严禁行为：**

- ❌ 一次性改完多个独立模块
- ❌ 只写页面骨架不做联调就宣告完成
- ❌ 页面功能未验证就更新为完成状态
- ❌ 让调试代码、原始 JSON 结果、临时注释长期留在页面里

---

## 开发节奏

### 1. 一次只推进一个清晰子目标

例如：

- 只做 `Overview` 配置页
- 只做 `Retrieve` 正式版
- 只做 `Chat` 正式版

不要在一个会话里同时重做多个大页面。

### 2. 先有设计，再写代码

如果涉及页面结构、用户流或模块边界变化：

- 先更新 `spec`
- 再更新 `plan/checklist/tasklist`
- 审核通过后再实现

### 3. 先有失败验证，再补实现

新增交互或修 bug 时：

- 优先补最小失败测试
- 再写最小实现
- 最后再重构

### 4. 页面完成不等于功能完成

一个页面只有满足以下条件才算完成：

- UI 可操作
- 请求真正打到后端
- loading / empty / error 有处理
- 类型检查通过
- lint 通过
- 至少有关键交互测试或浏览器联调验证

## 前端分层约束

### 路由层

`routes/` 只负责：

- 读取路由参数
- 组织页面容器
- 连接 feature 面板

不要把大量表单逻辑、mutation 逻辑、复杂布局都堆进路由文件。

### Feature 层

`features/` 负责：

- 业务组件
- 表单
- 局部 hooks
- schema

页面中的主交互逻辑优先放在 feature 层。

### API 层

`api/endpoints/` 负责：

- query hooks
- mutation hooks
- 成功提示
- query 失效与刷新

不要在页面组件里直接拼接裸请求。

### 状态层

- TanStack Query：服务端状态
- Zustand：纯 UI 状态

不要把 Query 的数据再复制一份进 Zustand。

---

## UI 约束

- 优先做正式产品态，不保留明显“调试页”观感
- 原始 JSON 返回不能作为正式交付界面长期存在
- 需要明确空态、无结果态、错误态、加载态
- 桌面工作台风格要统一，不同模块之间不要像多个拼装 demo

---

## 验证要求

每次提交前至少执行：

```bash
pnpm lint
pnpm check
pnpm test
```

如果改动涉及联调主链路，还应执行：

```bash
pnpm dev
```

并手动验证关键流程。

如果改动接近桌面端能力，还应进一步验证：

```bash
pnpm tauri:dev
```

---

## 提交要求

提交前必须保证：

- 本次目标清晰
- 代码可运行
- 没有遗留明显半成品
- 文档已同步更新

提交信息建议使用中文，并明确模块与目标，例如：

- `完善知识库配置工作台`
- `重构知识库检索调试页`
- `增加前端退出登录功能`

---

## 交接要求

如果一个模块没有做完，必须写清：

- 已完成什么
- 卡在哪
- 下一步应该从哪里接着做
- 哪些文件是这轮主要改动点

不要只说“还差一点”。

---

## 当前阶段优先级

当前前端优先级按以下顺序推进：

1. 知识库工作台
2. Agent Chat 页面正式版
3. 系统配置页体验完善
4. Tauri 桌面端收口

在知识库工作台完成前，不建议大规模切换到别的模块。
